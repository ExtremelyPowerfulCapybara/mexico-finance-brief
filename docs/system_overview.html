1. What the System Does

The Periphery is a fully automated newsletter pipeline that runs every weekday morning on a VPS. It fetches financial news, writes a bilingual digest using Claude, pulls live market data, sends an email to subscribers, and publishes a web archive page — all without human intervention.

The one exception: every morning, a collaborator reviews three AI-generated hero images in Telegram and selects one. That selection triggers the final render and site publish. Everything else is automatic.

What happens every morning

  1. 7:00 AM — Main pipeline runs Fetches news, calls Claude to write the digest, pulls market data, sends the email, builds the archive page.
  2. 7:06 AM — Image candidates generated Reads today's digest, generates 3 hero images via OpenAI, and sends them to Telegram for review.
  3. ~7:15 AM — You review Telegram Three candidate images appear with Select buttons. You pick one (or regenerate or skip).
  4. Within 2 minutes — Site updates The handler detects your selection, embeds the image in the archive page, and publishes the updated site.

2. End-to-End Pipeline

main.py — 7:00 AM (weekdays)

The main pipeline runs as a single Python process and completes in roughly 3–5 minutes.

StepWhat happens
Fetch market dataYahoo Finance queried in parallel: DXY, 10Y UST, VIX, MSCI EM, equities, commodities, crypto, and FX cross-rates for 8 currencies
Fetch newsNewsAPI queried across 7 Spanish topics, filtered to 14 curated outlets, deduplicated against the last 5 days
Score articlesEach article scored for freshness, source authority, and topic relevance; top 12 go to Claude
SummarizeClaude returns a bilingual JSON: 5–7 stories, editor note, narrative thread, sentiment score (5–95), quote of the day
Generate image promptsLead story tag and sentiment are mapped to 3 compositional image prompts — no API call yet, just text
Save digestFull JSON saved to digests/YYYY-MM-DD.json
Send emailGmail SMTP delivers the HTML newsletter to all subscribers
Build archivedocs/YYYY-MM-DD.html written; docs/index.html fully rebuilt
Telegram notificationLightweight message sent with today's lead headline and archive link

generate_candidates.py — 7:06 AM (weekdays)

Six minutes after main.py, this script picks up the digest and creates actual images.

StepWhat happens
Read digestLoads digests/YYYY-MM-DD.json to retrieve the 3 image prompts
Guard checkExits immediately if hero image already locked, or if candidate files already exist on disk
Generate imagesOpenAI Images API called 3 times; PNGs saved to tmp_images/YYYY-MM-DD/r1_opt1.png etc.
Update digestCandidate file paths written back into the digest JSON
Send to TelegramContext message sent, then each photo with a Select button, then Regenerate / Skip controls

Human interaction in Telegram

You receive the images and choose what to do:

Select

Pick this image for today's issue. Site updates automatically within 2 minutes.

Regenerate

Generate a fresh batch of 3 candidates. Maximum 2 regenerations per issue.

Skip

No image today. Archive page publishes without a hero. Always acceptable.

telegram_handler.py — every 2 minutes (all day)

This script polls Telegram continuously and handles each action:

ActionWhat the handler does
Select Copies selected image to docs/images/YYYY-MM-DD.png → verifies file → updates digest with hero_image → rerenders archive HTML → rsyncs docs/ to web root → deletes non-selected tmp files
Regenerate Increments round counter → generates 3 new images (round 2, 3…) → deletes previous-round tmp files → sends new candidates to Telegram
Skip Acknowledges; no changes to digest or site

4. Key Files

bot/main.py

Purpose: Entry point. Runs the entire morning pipeline in sequence.
When: Cron at 7:00 AM weekdays.
Reads: NewsAPI, Yahoo Finance, Anthropic Claude.
Writes: digests/YYYY-MM-DD.json, docs/YYYY-MM-DD.html, docs/index.html.

bot/generate_candidates.py

Purpose: Generates 3 hero image candidates and sends them to Telegram.
When: Cron at 7:06 AM weekdays.
Reads: digests/YYYY-MM-DD.json (specifically visual.hero_options).
Writes: tmp_images/YYYY-MM-DD/r1_opt*.png; updates the digest with candidate paths.
Idempotent: Safe to re-run — exits if candidates already exist or hero is locked.

bot/telegram_handler.py

Purpose: Polls Telegram for button presses and runs the corresponding action (copy, rerender, publish, regenerate).
When: Cron every 2 minutes around the clock.
Reads: Telegram API, digests/, tmp_images/.
Writes: docs/images/YYYY-MM-DD.png, digest JSON, docs/YYYY-MM-DD.html, web root via rsync.

bot/image_candidates.py

Purpose: Low-level image generation. Calls OpenAI Images API and saves the PNG.
When: Called by generate_candidates.py and on regeneration from telegram_handler.py.
Reads: OPENAI_API_KEY and image model settings from environment.
Writes: PNG files to tmp_images/.

bot/image_gen.py

Purpose: Builds image prompts (not images). Reads the lead story tag and mood from the digest, maps them to 3 compositional prompt variants. No API call.
When: Called inside main.py before the digest is saved.
Reads: Digest dict in memory.
Writes: Nothing — returns a visual metadata dict that gets saved inside the digest JSON.

bot/rerender.py

Purpose: Re-renders a single archive page from its saved digest. Used after image selection to embed the hero image in the HTML.
When: Called by telegram_handler.py after selection. Can also be run manually.
Reads: digests/YYYY-MM-DD.json.
Writes: docs/YYYY-MM-DD.html (overwrites).

bot/publish_site.py

Purpose: Rsyncs docs/ to the live web root.
When: Called by telegram_handler.py after rerender. Skips silently if PUBLISH_WEB_ROOT not set.
Reads: PUBLISH_WEB_ROOT from environment.
Writes: Destination directory at PUBLISH_WEB_ROOT.

6. Cron Jobs

Add these to crontab -e on the VPS. All times are Mexico City time.

CRON_TZ=America/Mexico_City

# Morning pipeline — weekdays at 7:00 AM
0 7 * * 1-5 cd /home/adrian/project/bot && /home/adrian/project/venv/bin/python main.py >> /home/adrian/project/logs/main.log 2>&1

# Image candidate generation — weekdays at 7:06 AM
6 7 * * 1-5 cd /home/adrian/project/bot && /home/adrian/project/venv/bin/python generate_candidates.py >> /home/adrian/project/logs/candidates.log 2>&1

# Telegram handler — every 2 minutes
*/2 * * * * cd /home/adrian/project/bot && /home/adrian/project/venv/bin/python telegram_handler.py >> /home/adrian/project/logs/telegram_handler.log 2>&1
Why the 6-minute gap matters: generate_candidates.py reads the digest that main.py writes. If it runs before main.py finishes, no digest exists yet and it exits silently. 6 minutes is a conservative buffer.

What happens when something fails

FailureEffectRecovery
main.py fails No digest, no email, no image candidates today Check logs/main.log; fix the issue; re-run manually
generate_candidates.py fails No images sent to Telegram today Re-run manually — it's idempotent and safe to retry
telegram_handler.py fails one cycle One 2-minute window missed Self-healing — next cycle picks up pending updates via offset

7. Daily Operation

The newsletter was already sent to subscribers at 7:00 AM. Your only job is to review and select the hero image.

Every morning checklist

  1. Open Telegram around 7:15 AM. You should see a context card followed by three candidate images.
  2. Read the lead headline shown in the context card. This is the story the image should support.
  3. Tap the Select button under the image that best matches the story's tone. You'll see a confirmation toast within a second.
  4. Done. The site updates automatically within 2 minutes.

When to use Regenerate

Use it when none of the three images feel right — the tone is off, they're too generic, or they don't match the lead story. You have 2 regenerations per issue before the button stops working.

When to use Skip

Use it when:

  • You don't have time to review today
  • None of the images are acceptable and you're out of regenerations
  • The story topic doesn't suit a visual

Skipping is always acceptable. The archive page publishes without a hero image and still looks correct.

8. Debugging Guide

No Telegram message received

Where to look: logs/candidates.log and logs/main.log

Likely cause: main.py failed before saving the digest, TELEGRAM_TOKEN or TELEGRAM_CHAT_ID not set, or OpenAI call failed.

Quick fix: Check both logs. If the digest exists, re-run manually:

cd /home/adrian/project/bot && python generate_candidates.py
Images not generated (context message appeared but no photos)

Where to look: logs/candidates.log

Likely cause: OPENAI_API_KEY missing or invalid, OpenAI rate limit, or malformed digest (no hero_options field).

Quick fix: Check logs/candidates.log for the error. Verify the digest has image prompts:

python -c "import json; d=json.load(open('digests/$(date +%F).json')); print(d.get('visual',{}).get('hero_options'))"
Selection not working (nothing happens after tapping Select)

Where to look: logs/telegram_handler.log

Likely cause: telegram_handler.py cron not running, PUBLISH_WEB_ROOT not set, or candidate file deleted before selection.

Quick fix: Run the handler manually to see live output:

cd /home/adrian/project/bot && python telegram_handler.py
Image not visible on the published site

Where to look: docs/images/, logs/telegram_handler.log

Likely cause: rsync failed, PUBLISH_WEB_ROOT not set, or path mismatch.

Quick fix:

  1. Confirm the PNG exists: ls docs/images/
  2. Check the digest's hero_image value is set correctly
  3. Manually publish: python publish_site.py
  4. If needed, rerender then publish: python rerender.py YYYY-MM-DD && python publish_site.py

9. Design Decisions

Why images are generated separately from main.py

Image generation takes 30–60 seconds per image (3 per issue = up to 3 minutes). Running it inside main.py would delay email delivery. Separating it into generate_candidates.py keeps the main run fast and makes the image step independently retryable if it fails.

Why Telegram for image selection

It's the lowest-friction editorial interface for a small team. No web dashboard to build or maintain. Button callbacks work on mobile. The only infrastructure needed is a bot token and a chat ID.

Why tmp_images/ vs docs/images/

Most candidate images are discarded. Only the selected image earns a permanent place in docs/images/. This keeps the published directory clean and avoids serving rejected images on the live site.

Why rsync for publishing

The VPS serves the site directly from PUBLISH_WEB_ROOT. rsync with --delete ensures the destination exactly mirrors docs/: new files are added, removed files are deleted, unchanged files are skipped. Fast, reliable, no framework needed.

10. Safety & Invariants

🔒
hero_image in the digest is a lock. Once set, generate_candidates.py and telegram_handler.py exit immediately. Do not clear it manually unless you intend to re-run the entire image flow.
⚠
Do not edit digest JSON while telegram_handler.py is polling. The handler reads and writes the same file. Concurrent edits can corrupt it. Stop the cron or wait until the handler is idle before editing manually.
✅
tmp_images/ can be safely deleted. These are temporary files. If deleted after selection, nothing breaks — the selected image is already in docs/images/. If deleted before selection, re-run generate_candidates.py.
📷
docs/images/ is the source of truth for published images. The archive HTML references /images/YYYY-MM-DD.png. Do not rename or move files here without also updating the digest and rerendering the archive page.
📄
Do not delete files from digests/. The archive index is rebuilt from all digest files on every run. Deleting a digest removes that issue from the historical record permanently.
🔄
docs/index.html is auto-generated. It is overwritten on every run. Any manual edits will be lost at the next run.
